設計方針は、**レビュー編集画面 S3 は dynamic 情報（基本/見た目/香り/味/その他）を扱う画面**であり、設定画面 S7 で表示モード等を切り替える前提に合わせています。
また、M3/Android 公式では、関連項目のグルーピングは **white space と typography を基本**にし、divider や card のような explicit containment は必要時のみ使うのが基本です。M3 の標準タイポグラフィは `titleMedium 16/24`、`titleSmall 14/20`、`bodyLarge 16/24`、`bodyMedium 14/20`、`bodySmall 12/16` です。

# Codex 指示書: レビュー編集画面（S3）UI改善

## 目的

レビュー編集画面（S3）の視認性と情報階層を改善する。
現状は、下位グループ（例: テクスチャ、味わい、上立香）を divider や card で表現しようとしているが、境界表現が強すぎてくどく見える。
この改善では、**Material Design 3 に沿って、グルーピングを「枠」ではなく「タイポグラフィ + 余白」で表現する**。

## 基本方針

* 下位グループの表現に **Card を使わない**
* 下位グループの表現に **Divider を常用しない**
* グルーピングは **subheader（小見出し）+ vertical spacing** で作る
* 入力コンポーネント自体が十分強いので、周辺の装飾は弱くする
* 画面全体の横幅はそろえる
* 既存のタブ構成（情報 / 見た目 / 香り / 味 / 特記）は維持する
* 既存の5段階選択UIは基本維持し、周辺レイアウトのみ整理する
* 画面は縦スクロール前提
* 下部の保存ボタンは維持する

---

## デザインルール

### 1. グルーピングルール

* 大分類はタブで表現済みなので、タブ内では大分類見出しを乱用しない
* 下位グループは **subheader** で表現する
* subheader の直下に、そのグループに属する項目を並べる
* グループ境界は **余白差** で見せる
* Divider は原則使わない
* どうしても必要な場合のみ、**大きな意味の切替箇所に1本だけ inset divider** を使う

### 2. Typography

| 用途                          | Token      | Size / Line height | Weight  |
| --------------------------- | ---------- | -----------------: | ------- |
| 画面タイトル（AppBar）              | titleLarge |        22sp / 28sp | Medium  |
| タブ直下の補助文（`* は必須項目です`）       | bodySmall  |        12sp / 16sp | Regular |
| 下位グループ見出し（例: テクスチャ、味わい、上立香） | titleSmall |        14sp / 20sp | Medium  |
| 各項目ラベル（例: 丸さ、酸味、強さ、具体例）     | bodyMedium  |        14sp / 20sp | Regular |
| 補助説明・ヘルプ                    | bodySmall  |        12sp / 16sp | Regular |
| クリア                         | bodyMedium |        14sp / 20sp | Regular |

### 3. Spacing

画面内余白は 8dp グリッドで統一する。

| 箇所                               |    値 |
| -------------------------------- | ---: |
| 画面左右余白                           | 16dp |
| タブ下の補助文 上                        | 16dp |
| タブ下の補助文 下                        | 12dp |
| subheader 上（通常）                  | 24dp |
| subheader 上（直前がsubheaderグループの場合） |  0dp |
| subheader 上（画面先頭の最初のsubheaderのみ） |  0dp |
| subheader 下                      | 4dp |
| 項目ラベル → 入力UI                     |  8dp |
| 同一グループ内の項目間                      | 16dp |
| グループ末尾 → 次グループ見出し                | 24dp |
| グループ末尾 → 通常項目                    | 36dp |
| 最後の入力UI → 保存ボタン領域                | 24dp |

### 4. Color

* subheader は `onSurface` または `onSurfaceVariant` 系で表現する
* 入力UIの緑系アクセントと競合しないように、subheader には強いアクセントカラーを使わない
* 枠線や背景色ブロックは追加しない
* 画面背景は既存テーマを尊重する

## レイアウト仕様

### 味タブ

#### 構造

1. `* は必須項目です`
2. `アタック`
3. アタック入力UI
4. `テクスチャ`（subheader）
5. `丸さ`
6. 丸さ入力UI
7. `舌ざわり`
8. 舌ざわり入力UI
9. `味わい`（subheader）
10. `甘味`
11. 甘味入力UI
12. `酸味`
13. 酸味入力UI
14. `苦味`
15. 苦味入力UI
16. 必要なら以降の項目（旨味、甘辛度、含み香、余韻、複雑性 など）も同ルールで配置

#### 重要ルール

* `アタック` は単独項目なので subheader ではなく通常ラベル扱い
* `テクスチャ` と `味わい` は subheader 扱い
* Divider は削除する
* Card で囲わない
* `クリア` は各項目行の右端に置くが、**未選択時は非表示または低強調表示**にする

### 香りタブ

#### 構造

1. `* は必須項目です`
2. `上立香`（subheader）
3. `強さ`
4. 強さ入力UI
5. `具体例`
6. 具体例入力UI（プルダウン/選択UI）
7. `複雑性`
8. 複雑性入力UI

#### 重要ルール

* `上立香` は subheader
* `複雑性` は独立した通常項目として扱う
* `複雑性` の前後に divider を入れない
* outlined の入力UIがある場合も、その前後の余白で整理し、追加の囲いは作らない

---

## コンポーネント単位の仕様

### 項目ブロック

各入力項目は次の順で構成する。

1. 項目ラベル行

   * 左: 項目名
   * 右: `クリア`（未選択なら非表示または低強調）
2. ラベル下 8dp
3. 入力UI
4. 次項目まで 16dp、または次グループなら 24dp

### subheader ブロック

1. 上余白 24dp
   ただし、**画面先頭の最初のsubheader**、または**直前がsubheaderグループの場合**は 0dp
2. `titleSmall`
3. 下余白 4dp
4. 配下項目群
5. グループに属する項目の配置完了後、後続の通常項目との間に 36dp、次グループとの間に 24dp の大きい余白を置く

---

## 禁止事項

* 下位グループを Card で囲う
* 下位グループごとに背景色ブロックを敷く
* 下位グループごとに枠線を付ける
* 項目ごとに divider を入れる
* subheader をアクセントカラーで強く目立たせる
* 横幅が項目ごとにズレるレイアウトにする

---

## 実装上の補足

* 既存の 5 段階選択UI コンポーネントは再利用してよい
* UI改善が目的なので、DBスキーマ変更はこのタスクのスコープ外
* 既存の mode visible 制御がある場合、それは維持する
* 今回は画面構造整理が主目的であり、コンポーネントの再発明は不要
* 余白値・タイポグラフィ token は定数化し、タブ間で共通利用できるようにする

---

## 受け入れ条件

* 味タブと香りタブから Card 表現が消えている
* 味タブと香りタブから不要な Divider が消えている
* グルーピングが subheader と余白だけで視認できる
* 最初のsubheaderの上に不自然な余白がない
* 画面の左右余白が統一されている
* 項目ラベル、subheader、補助文の文字階層が明確
* 見た目が M3 に寄り、以前より軽く整って見える
* 既存機能（入力、保存、タブ切替）は壊さない

---

## 実施タスク

1. レビュー編集画面のレイアウト構造を整理
2. Divider/Card の削除
3. subheader 用 composable またはスタイル定義を追加
4. 項目ブロックの spacing ルール統一
5. `クリア` 表示条件の見直し
6. 味タブ・香りタブに適用
7. 必要なら見た目タブ・特記タブにも同じルールを横展開
8. スクリーンショット比較で before/after を確認

---

## 実装方針

* まず `ReviewSectionSubheader` と `ReviewFieldBlock` のような共通部品に分離する
* 次に味タブと香りタブのレイアウトをその部品で置き換える
* その後、spacing token を整理する
* 最後に `クリア` の表示条件を微調整する

---

必要なら次に、これを **Compose 前提のさらに実装寄りプロンプト** に落として、`Composable` 名や `Modifier.padding()` の粒度まで書きます。
